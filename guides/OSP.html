OSP at a Glance

OVERVIEW

📋 What is it?

An externally set, internally assessed synoptic project. You respond to a given scenario and are assessed on your application of the six Performance Outcome skills. Externally marked by Pearson.

🎓 Grading

The OSP is graded: Distinction, Merit, Pass, or Unclassified. You must achieve at least a Pass to be certificated for the T Level.

Functional vs Non-Functional Requirements

PO1 · ANALYSE
"Secure by design" — security must be considered from the start and must influence both functional and non-functional requirements. It is not an afterthought.

Functional vs Non-Functional

✅ Functional Requirements

WHAT the system does — its features, behaviour, and capabilities.

  • The inputs required
  • The data needed
  • The data processing that must take place
  • The logic of the system
  • The deployment and usage platforms

Example: "The system must allow users to reset their password via email."

⚙️ Non-Functional Requirements

HOW WELL the system performs — quality attributes and constraints.

  • Security considerations
  • Required accessibility features
  • Scalability requirements
  • KPIs: responsiveness, load handling, reliability
  • User acceptance criteria

Example: "The system must respond in under 1.5 seconds for 95% of requests."

User Centred Design (UCD) — Iterative Cycle

1️⃣ Understand

Understand the use context — who, what, when, how, why

2️⃣ Specify

Specify user requirements — what the user needs to achieve

3️⃣ Design

Design the solution to meet those requirements

4️⃣ Assess

Assess the design against requirements → loop back

PO2: Design, Implement & Test

PO2 · DESIGN & BUILD
🏆 57 marks · 39.6% — The highest-weighted PO. You design, build, and test the software using appropriate approaches, languages, and testing techniques.

Design Approaches You Must Know

Top-Down / Functional

System built as sub-functions. Data flows through the system. Each function handles or transforms data. Uses data flow diagrams.

Object-Oriented (OOP)

Code is reusable through classes, methods, objects. Four pillars: Encapsulation, Abstraction, Polymorphism, Inheritance.

Test-Driven (TDD)

Write a failing test → write code to pass it → refactor. Ensures every feature has a test. Common cycle: add test → run → write code → run → refactor.

Behaviour-Driven (BDD)

Development informed by required behaviour specified before coding. Behaviours must have business value. Structure: title, narrative, acceptance criteria.

Functional Design

Program structured as modules performing a single defined process. Characteristics: recursion, closures, first-class functions, higher-order functions.

Data Model Design

Visualise how data is organised. Use ERDs and UML. Common models: conceptual, logical, physical, hierarchical, relational.

Task 1 Activity C — AI Code Generation

PO2 · DESIGN & BUILD
🤖 This is unique to the OSP — you are required to use a generative AI model (e.g. ChatGPT, Google Gemini) to produce specific code snippets, then review and integrate them into your larger solution.

What the Spec Requires

✅ You MUST Do

  • Use AI to create short snippets for specific functions or processes
  • Demonstrate the specific prompts you entered
  • Review the AI output critically — does it meet requirements?
  • Refine the output as required before integrating it

❌ You Must NOT Do

  • Use AI to generate the entire solution in one prompt
  • Accept AI output without reviewing it against requirements
  • Use AI to replace your own design work
Why this is assessed: The spec wants to see you can use AI as a professional tool — writing precise prompts, evaluating output critically, and integrating it appropriately. Uncritical copy-paste from AI is not evidenced competence.

What to Document

📝
Show your prompts — screenshot or paste the exact prompt you used. Show the output — what did the AI produce? Show your review — what did you accept, change, or reject, and why?

PO3: Change, Maintain & Support

PO3 · MAINTAIN
🔧 18 marks · 12.5% — Understand what drives change in software and how to manage the change process professionally.

Why Software Changes

🛡 Preventative Changes

  • Changes to regulatory requirements
  • Compatibility with new products or technology
  • Changes in business process
  • Release of a new product or service

🚨 Corrective Changes

  • New vulnerabilities (zero-day attacks)
  • Targeted attack or data corruption
  • System failures

🔄 Iterative Development

  • User/client feedback
  • New technology developments
  • Competition
  • Need to improve efficiency
  • Future-proofing

Change Management Process

1️⃣
Identify issues/changes from feedback/review process
2️⃣
Document developments and changes for technical and non-technical audiences
3️⃣
Plan and schedule the required changes
4️⃣
Regression test — verify changes didn't break existing functionality
5️⃣
Release the updated product — planned or reactive

Supporting Users & Maintaining Code

PO3 · MAINTAIN

Systematic Support Process

StepWhat to do
1. Identify / ReplicateReproduce the issue to confirm it exists and understand the circumstances
2. Investigate CauseDetermine if it's a user error, system error, application error, or security breach
3. Apply TestingIdentify errors in code; make changes; ensure error doesn't return; check no new issues
4. Communicate ResolutionTell appropriate stakeholders how and when the issue was resolved
5. Document LessonsRecord what happened and what was learned to prevent recurrence

Good Code Maintenance Principles

Code Quality

  • Separate code and use modularity
  • Readability above all
  • Consistent naming conventions
  • Informative commenting and annotation

Documentation

  • Update change logs after every change
  • Write for both technical and non-technical audiences
  • Release notes, user guides, FAQs, API contracts

Legal & Regulatory Considerations

Legal Requirements

Ethical Considerations

  • Software licensing

Risk Management — CIA Triad & Policies

🔒 Confidentiality

Data accessible only to those authorised. Encryption, access controls, authentication.

✅ Integrity

Data is accurate and unmodified. Hash checks, audit trails, version control.

🟢 Availability

Data and systems are accessible when needed. Backups, disaster recovery, uptime monitoring.

🗺️
Risk assessment process: Identify risk → Assess likelihood and seriousness → Consider potential impact → Identify mitigations → Contingency planning → Ongoing monitoring

Task 3a & 3b: Feedback & Evaluation

PO3 & PO5
Task 3a (internet permitted) — Gather feedback from users. Task 3b (no internet) — Evaluate that feedback and write a reflective evaluation.

Task 3a: Gathering Feedback

Methods

  • User observation with observation records
  • Surveys and questionnaires (Google Forms, SurveyMonkey)
  • Focus groups representative of the target audience
  • Individual interviews
  • Verbal feedback (formal and informal)
  • Performance and usage data

What to Capture

  • Does the product meet functional requirements?
  • Does it meet user needs and acceptance criteria?
  • Are there usability issues?
  • What do users suggest for improvement?

Task 3b: Evaluating Feedback (No Internet)

📊
Analyse patterns in the feedback — what did multiple users agree on? Evaluate against requirements — which criteria were met or not met? Recommend improvements — specific, evidenced, justified.
Distinction tip: Don't just report what users said — evaluate its significance. "Three users found navigation confusing. This is significant because usability is a non-functional requirement — I would address this by implementing [specific design change]."
