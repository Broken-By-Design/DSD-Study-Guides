Task 1: Planning a Project

TASK 1 · PLANNING
🗂️ AO1 — Planning (12% of total marks) You use a provided template to create a complete project plan, including a Gantt chart, a resource and cost plan, and a written rationale.

Three Deliverables

📅 Gantt Chart

A visual schedule showing tasks, durations, dependencies, team member assignments, and milestones across the project timeline.

💰 Resource & Cost Plan

Lists all resources (people, equipment, materials) with associated costs. Must show total project cost and assess affordability.

✍️ Rationale

A written justification of the key decisions made in your plan — order of tasks, team allocation, risk considerations, cost impacts.

SDLC Connection

Task 1 maps to the Planning & Requirements Analysis lifecycle stage. You must use an appropriate software development lifecycle (Agile, Waterfall, RAD, Lean, Scaled Agile) to organise your project tasks efficiently.

Resource & Cost Plan

TASK 1 · PLANNING
📋 Purpose: Identify and calculate all project costs, then determine whether the project is affordable and viable for the client over time.

Three Cost Categories

🧱 Materials

Software licences, development tools, frameworks, APIs, cloud service subscriptions, stock images/assets used in the project.

💻 Physical Resources

Hardware, servers, workstations, networking equipment, testing devices (mobile, tablet). Include depreciation where relevant.

👥 Personnel

Team members' time (day rates or hourly rates × hours allocated). Include project manager, developers, testers, UX designers.

Beyond the Numbers

Affordability

Can the client actually pay for this? Compare total project cost against client budget. Identify any cost savings or value-engineering options.

Viability Over Time

What are the ongoing costs after go-live? Maintenance, support, licence renewals, server hosting. Does the solution provide value long-term?

Exam tip: To hit higher marks, don't just list costs — justify them. Explain why you chose a particular resource and what its cost impact is on the project timeline.

Rationale

TASK 1 · PLANNING
✍️ The rationale is your written justification of the most significant planning decisions you made. It shows the examiner that you've thought critically, not just filled in a template.

Four Areas to Justify

⏱ Order & Timing of Tasks

Why did you schedule tasks in this order? What dependencies drove your decisions? How did you handle overlapping work?

👥 Allocation of Team Members

Why did you assign specific people to specific tasks? How do their skills match the task requirements?

⚖️ Benefits & Risks

What are the potential risks of your plan? How have you mitigated them? What are the benefits of your approach?

💸 Impact on Timings & Costs

How do your decisions affect the overall timeline and budget? What trade-offs did you make?

📝 High-mark tip: Use connective phrases like "I assigned [person] to [task] because their experience in X means..." and "This scheduling decision reduces risk because..." — examiners reward explicit, reasoned justification.

Task 2: Testing & Debugging

TASK 2 · TESTING
🐛 AO2 — Application. You are given existing code with defects. You must assess it against the requirements, carry out testing, document the process, and fix the errors.

Three Parts to Task 2

1. Identify Defects

Assess the given code against its requirements. Run it, read it, and identify where it fails.

2. Document Testing

Create annotated testing evidence: purpose, test data, expected results, actual results, comparison, and further actions.

3. Fix the Code

Correct errors and add/remove code to make it functional. Follow programming conventions and justify every decision.

Programming Conventions Required

Precise logic Correct indentation Meaningful variable names Consistent commenting Correct output
Remember: You must justify decisions made when fixing defects. Don't just change the code — explain in your annotations WHY you made each change.

Fixing the Code

TASK 2 · TESTING
The solution: Correct errors and add/remove code to ensure the given code is fully functional and meets the stated requirements.

Common Defect Types

Syntax Errors

Code that violates the rules of the language (e.g. missing colon, wrong indentation in Python, mismatched brackets). The program won't run at all.

Logic Errors

Code that runs but produces wrong output (e.g. > instead of >=, wrong calculation, loop runs too many/few times).

Runtime Errors

Code that crashes during execution (e.g. dividing by zero, accessing an index that doesn't exist, calling a function that doesn't exist).

Requirement Errors

Code that runs without errors but does not meet the specification (e.g. wrong output format, missing validation, incorrect feature).

Good Fixing Annotations Include

💬
Before: Quote the original buggy line or section. After: Show your corrected version. Why: Explain your reasoning — "I changed < to <= because the boundary value 0 should be accepted as valid."
Defensive programming tip: When fixing, also consider edge cases. Add input validation, exception handling, and meaningful error messages where the original code lacked them.

Task 3: Designing a Solution

TASK 3 · DESIGN
🎨 AO2 + AO3. You break down the problem, design algorithms, follow conventions, and produce documentation detailed enough for a third party to build the solution.

Three Design Requirements

1. Decomposition

Break the problem into smaller computational parts. Use abstraction, show reusable components, identify all sub-systems.

2. Algorithm Design

Describe each part using algorithms. Must include sequence, selection and iteration. Use flowcharts or pseudocode with correct conventions.

3. Design Communication

Produce documents clear enough for the client to make decisions AND for a third party to build the solution from.

Design Tools You Might Use

Flowcharts (BCS symbols) Pseudocode Wireframes ERD / Data Dictionary UML Class Diagrams Data Flow Diagrams
💡 Key: Use a combination of written and diagrammatical presentation. Pure text or pure diagrams alone will limit your marks.

Decomposition & Algorithms

TASK 3 · DESIGN

Decomposition Principles

Break It Down

  • Identify all sub-systems that make up the solution
  • Show how sub-systems connect and interact
  • Each part should be solvable computationally
  • Justify decisions about how you split the problem

Abstraction & Reuse

  • Remove unnecessary detail — focus on what matters
  • Identify reusable components (functions, modules)
  • Make effective use of detailed abstraction and refinement

Algorithm Design Requirements

RequirementWhat it meansExample
Clearly define stepsEvery step unambiguously specifies an actionSet total = 0, not "add up numbers"
Uniquely definedEach step depends on input and result of previous stepsSteps follow logically, no ambiguity
SequenceSteps execute in orderInput → Process → Output
SelectionConditional branching (IF/ELSE)IF age >= 18 THEN…
IterationRepeating a block of code (FOR/WHILE)WHILE balance > 0…
Flowchart reminder: Use correct BCS symbols — parallelogram for I/O, rectangle for process, diamond for decision, oval for start/end. Incorrect symbols lose marks.

Design Communication

TASK 3 · DESIGN
📄 Threshold: Your design documents must be detailed enough that a third party who has never seen the brief could use them alone to build the proposed solution.

Three Audiences to Satisfy

👤 The Client

Needs enough detail to make informed decisions about the solution. Use plain language and diagrams they can understand.

🔨 Third Party Developer

Must be able to build the solution from your documentation alone. Needs precise technical detail, data structures, and algorithm logic.

📋 Examiner

Assesses whether your design is clear, complete, and professional. Rewards use of technical vocabulary and consideration of audience.

What Good Design Documentation Contains

Written Elements

  • Explanation of structures and processes
  • Appropriate technical vocabulary
  • Justification of design decisions
  • Consideration of the audience

Diagrammatical Elements

  • Flowcharts using standard BCS symbols
  • Pseudocode with correct notation
  • Wireframes / UI mockups
  • ERDs, class diagrams, data flow diagrams

Task 4a: Developing a Solution

TASK 4A · DEVELOPMENT
⚙️ AO2 + AO5. Apply your programming skills to develop a working solution that meets the brief requirements. Use your Task 3 design as the blueprint.

What the Spec Requires

Technical Skill

  • Demonstrate appropriate programming techniques
  • Use pre-written and user-written modules
  • Appropriate use of interfaces between modules
  • Refine the system as required during development

Problem Solving

  • Show understanding of programming concepts
  • Apply computational thinking (decomp, abstraction)
  • Handle edge cases and unexpected input
  • Produce consistently correct outcomes
Note: Formal test documentation is NOT required in Task 4a — but testing should still take place implicitly. Evidence of testing is a working product that meets the brief requirements.

Languages You Can Use

Python 3.10+ C# SQL JavaScript PHP

Code Organisation

TASK 4A · DEVELOPMENT
💻 The specification lists explicit code quality requirements. Meeting these separates a pass from a merit/distinction.

Required Code Quality Principles

Structure

  • Clear, meaningful indentation throughout
  • Avoid deeply nested if-clauses and loops
  • Use functions/classes/objects with proper structure
  • Consistent style throughout the whole solution

Documentation

  • Comments wherever possible to explain logic
  • Meaningful variable, function, and class names
  • Use of constants for fixed values

Variables

  • Good use of local variables
  • Minimal use of global variables
  • No copy-pasted procedural code

Security & Robustness

  • Defensive programming throughout
  • Handle data securely
  • Good exception handling (try/except)

User Experience Requirements

🖥️
Consistency · Simplified user input · Meaningful output messages · Explanation of what to input · Error messages · Data visualisation (e.g. Matplotlib graphs)

Task 4b: Reflective Evaluation

TASK 4B · EVALUATION
📝 AO5b — Review (9% of total marks). Apply reflection and evaluation techniques. Show you can honestly assess your own work and identify improvements.

Three Parts of the Evaluation

1. Evidence of Success

Demonstrate that your product meets the brief requirements. Include specific measures against the success criteria listed in the brief.

2. User Needs

Provide evidence from testing that the product meets user needs. Reference actual test results and user feedback where possible.

3. Improvements

Discuss how the solution could be improved if you revisited the problem. Be specific — say what you would do differently and why.

High-Mark Evaluation Language

✅ Do Say

  • "The product meets criterion X because [evidence]…"
  • "Testing showed that when [input], the output was [result], which confirms…"
  • "If I revisited this, I would improve [X] by [specific action] because…"

❌ Avoid

  • Vague statements: "I think it went well"
  • Listing features without linking to the brief
  • "I would add more features" without specifics
Reflection vs Evaluation: Reflection asks "what happened?" — Evaluation asks "how well did it meet requirements and what could be better?" Aim for evaluation, not just description.
