PROBLEM SOLVING · 1.1

Computational Thinking

Definition (1.1.1): A structured, logical approach to problem-solving that enables complex problems to be broken down into forms a computer can help to solve. It is a way of thinking applicable across all areas of software development — not just coding.

When to use (1.1.2)

  • When facing a large or complex problem difficult to solve in one step
  • When designing a new software system or algorithm from scratch
  • When analysing an existing system to find improvements or fix defects
  • When communicating a solution to a technical or non-technical audience

Benefits & Drawbacks (1.1.3, 1.1.5)

BenefitsDrawbacks
Breaks complex problems into manageable partsTime-consuming to apply initially
Encourages reuse of existing solutionsRisk of over-simplification losing context
Makes collaboration and communication clearerRequires practice to become natural
Improves efficiency and correctness of codeAbstraction may hide needed details later
Applicable to any programming language or paradigmDecomposition may miss cross-cutting concerns

The Four Components (1.1.4)

🔧 Decomposition

Breaking a large, complex problem and its solution into smaller, more manageable sub-problems that can each be solved, developed and tested independently before being combined.

🔍 Pattern Recognition

Finding and interpreting trends, similarities and repeated structures within or between problems — enabling previously known solutions to be adapted and reused without starting from scratch.

🎭 Abstraction

Identifying what information is essential to solving the problem and filtering out unnecessary detail — hiding internal complexity so developers and users only see what they need to.

📐 Algorithmic Design

Creating a precise, unambiguous, step-by-step set of instructions — using sequence, selection and iteration — that can be followed by a human or computer to solve the problem.

💡Interrelationships (1.1.17): Used together: decompose first → recognise patterns across sub-problems → abstract away irrelevant detail → design the algorithm. Exam questions ask you to judge which component is most suitable for a given scenario and why.
PROBLEM SOLVING · 1.1

Decomposition

Purpose (1.1.6): To make a complex problem manageable by dividing it into sub-problems that can each be understood, developed and tested separately, then combined into a final solution.

Tasks of Decomposition (1.1.7)

  1. Identify the main features of the problem — what is it asking you to produce, do, or solve?
  2. Characterise each identified feature — what inputs does each part need? What outputs does it produce? What data does it use?
  3. Break the problem down into smaller, more manageable parts — each part should be solvable without needing to understand the whole.
  4. Break the solution down into sub-solutions or modules that can be developed and tested independently before integration.
Example — Grade Management System: (1) Accept student name and marks as keyboard input → (2) Validate all inputs (presence, range, type) → (3) Calculate average mark → (4) Determine grade band using selection → (5) Display result to screen → (6) Append result to text file.

PROBLEM SOLVING · 1.1

Pattern Recognition & Abstraction

🔍 Pattern Recognition — Purpose (1.1.11)

To find and interpret similarities, trends and common features within and between problems, enabling reuse of known solutions and informed predictions about unknowns.

Using Pattern Recognition (1.1.12)

  • Find and interpret trends and similarities within and between problems and processes — e.g. login validation follows the same pattern as any form input validation
  • Find and interpret common features between a given problem and existing solutions — e.g. a leaderboard is a sorted list with a display loop; reuse the sort function
  • Make predictions and assumptions based on identified patterns — e.g. if an O(n²) sort takes 1ms for 100 items, it will take ~4ms for 200 items (quadratic growth)
💡 Example: A developer recognises that calculating the average of exam marks, sensor readings, and daily sales figures all follow the same pattern: sum all values ÷ count. One reusable calculate_average(values) function serves all three use cases.

🎭 Abstraction — Purpose (1.1.13)

To simplify a problem by focusing on what is essential — hiding unnecessary details of internal workings so the solution can be understood and used without knowing its implementation.

Tasks of Abstraction (1.1.14)

  • Identify information that is needed — what data is actually required to solve the problem?
  • Filter out unnecessary details — ignore irrelevant attributes that do not affect the solution
  • Hide details of internal workings — expose only the interface (inputs/outputs), not the implementation

When using abstraction, identify: (1.1.15)

What inputs are needed Expected outputs & outcomes Things that will vary Things that remain constant Key actions the solution must perform Repeated processes the solution will perform
Abstraction example — Login system: Inputs: username (str), password (str). Output: authenticated (bool). Constants: MAX_ATTEMPTS = 3. Varies: current attempt count. Key action: compare password hash to stored hash. Hidden: exact hashing algorithm (SHA-256), database connection details. The caller only needs to know: call login(user, pw) → returns True/False.
⚠️Exam tip — Interrelationships (1.1.17): You may be asked to "make judgements about the suitability" of using specific CT components. Consider: Is the problem complex enough to need decomposition? Are there similar problems whose patterns can be reused? What details can safely be abstracted away without losing correctness?
PROBLEM SOLVING · 1.2

Algorithmic Design

Definition (1.2.1): An algorithm is a finite, ordered, unambiguous set of instructions that takes defined inputs, processes them through clear steps, and produces defined outputs. Every algorithm must eventually terminate.

Expressing Algorithms — Benefits & Drawbacks (1.2.2–1.2.5)

MethodBenefitsDrawbacks
FlowchartsVisual; traces execution path; shows branching and loops; accessible to non-programmers; clear decision pointsLarge/cluttered for complex logic; symbols must be used correctly; hard to represent nested loops clearly
Written descriptionsEasy to write; understandable by clients; can convey intent not just steps; no special software neededAmbiguous language; cannot be executed; no standard enforced; different readers may interpret differently
Code (Python)Directly executable; unambiguous; immediately testable; uses spec keywords (Appendix 2)Requires Python knowledge to read; implementation detail can obscure the high-level logic

Control Structures (1.2.6)

StructurePurposePython syntax
SequenceExecute steps one after another in written order — the default(no keyword)
SelectionChoose which path to execute based on a conditionif, elif, else, match/case
IterationRepeat a block of code multiple timesfor (count-controlled), while (condition-controlled)

Algorithm Skills (1.2.7–1.2.12)

  1. Determine purpose (1.2.7) — read an algorithm and explain in plain English what it does and why
  2. Determine output given input (1.2.8) — trace through step-by-step, tracking variable values in a trace table
  3. Identify errors (1.2.9) — find logic errors (wrong output), syntax errors (won't run), or boundary errors (fails at edge cases)
  4. Correct errors (1.2.10) — fix the identified error without introducing new bugs
  5. Translate between notations (1.2.11) — convert flowchart → written description → code and back
  6. Design algorithms (1.2.12) — write new algorithms using all three control structures appropriately
# Trace table: find_max([3, 7, 2, 9, 1])
# Iteration | n  | max_val | n > max_val?
#     —     | —  |    3    | — (init)
#     1     | 3  |    3    | No
#     2     | 7  |    7    | Yes → update
#     3     | 2  |    7    | No
#     4     | 9  |    9    | Yes → update
#     5     | 1  |    9    | No
# Returns: 9
def find_max(numbers):
    max_val = numbers[0]        # sequence: initialise
    for n in numbers:            # iteration: count-controlled
        if n > max_val:          # selection: condition check
            max_val = n
    return max_val
PROBLEM SOLVING · 1.3

Problem-Solving Strategies

⬇️ Top-Down (1.3.1)

Start with the overall problem at the highest level and progressively break it into smaller sub-systems until each component is small enough to implement directly.

Benefits
Clear overall structure visible from the start
Easy to divide work across a large team early
Stakeholders can understand the system design before coding begins
High-level documentation produced naturally
Drawbacks
Lower-level implementation challenges only discovered late
Risk of over-engineering before full detail is known
Hard to adapt if high-level design needs changing

⬆️ Bottom-Up (1.3.1)

Build and test small, fundamental components first (individual functions/modules), then progressively combine them into larger, more complex sub-systems and eventually the complete solution.

Benefits
Components are tested and verified reliable before integration
Encourages reusable, modular, well-tested code
Practical implementation challenges identified early
Good for libraries of reusable components
Drawbacks
Can lack overall architectural vision at the start
Integration of components can be complex
Risk of building components that don't fit together well

🧩 Modularisation (1.3.1)

Divide the solution into independent, self-contained modules, each performing a single, well-defined responsibility with a clear interface (what goes in and what comes out).

Benefits
Modules are independently testable in isolation
Teams can develop different modules in parallel
Modules can be reused across multiple projects
Maintenance is easier — change one module without affecting others
Drawbacks
Requires careful upfront interface design
More planning overhead and documentation needed
Interface changes can cascade across the system

High-Level Problem-Solving Process (1.3.5)

StepWhat it involves
1. Define the problemClearly state exactly what needs to be solved — the requirements. Avoid assumptions. Involve stakeholders.
2. Gather informationResearch the problem domain, users, constraints, regulations, existing solutions and available data.
3. Analyse the informationIdentify patterns, constraints, risks, root causes and the core sub-problems to be solved.
4. Make a plan of actionChoose a strategy, design the algorithm or system architecture, identify resources and set a realistic timeline.
5. Implement a solutionWrite the code following the plan, applying good coding practices and conventions throughout.
6. Review the solutionTest thoroughly against requirements, gather stakeholder feedback and identify areas for improvement.
⚠️ Exam tip — Making judgements (1.3.6): Questions ask you to evaluate which strategy suits a given scenario. Consider: How well-understood is the problem? How large is the team? Are reusable components available? Is there a fixed deadline? Is iterative development expected? Modularisation works with both top-down and bottom-up approaches.
PROBLEM SOLVING · 1.3

Root Cause Analysis

Purpose (1.3.3): Root Cause Analysis (RCA) is a systematic process used to identify the underlying, fundamental cause of a problem or defect — rather than just treating the visible symptom. Without finding the root cause, the same problem will recur.

❓ Five Whys (1.3.4)

Ask "Why?" five successive times, where each answer becomes the basis of the next question. The fifth answer is typically the root cause. Best for simple, linear cause-effect chains.

Worked Example:
Problem: Users cannot log in
Why 1? The database connection fails
Why 2? The connection pool is exhausted
Why 3? Connections are not released after use
Why 4? The error-handling code was accidentally removed
Why 5? The code review process was not enforced
✓ Root cause: Code review not enforced
Fix: Mandate peer review on all commits

📊 FMEA (1.3.4)

Failure Mode and Effects Analysis — used proactively during the design phase to systematically identify all possible ways a system could fail, assess the likelihood and impact of each failure, and plan preventative actions before deployment.

StepWhat you do
1. List componentsIdentify every part of the system
2. List failure modesHow could each component fail?
3. Assess effectsWhat happens when it fails?
4. Rate severity (1–10)10 = catastrophic failure
5. Rate likelihood (1–10)10 = will definitely occur
6. MitigationActions to reduce risk score

Best for: Complex systems where multiple independent failure modes must all be addressed proactively before launch.

🌳 Event Tree Analysis (1.3.4)

ETA is a forward-looking graphical method. Starting from one initiating event, it maps all possible outcomes by showing success/failure branches at each subsequent stage of the system's response.

Example: Starting event: "Server receives unexpected input" → Branch 1: Does firewall block it? (Yes → safe; No → continue) → Branch 2: Does input validation catch it? (Yes → error logged; No → continue) → Branch 3: Does database query use parameterisation? (Yes → safe; No → SQL injection).

Best for: Analysing consequences of a specific event after it occurs, or anticipating consequences before deployment.

Actions After RCA (1.3.4)

ActionWhen
LogAlways — record issue, analysis, findings, actions taken
CloseRoot cause fixed and fix verified — mark as resolved
EscalateBeyond your expertise or authority — pass to manager, specialist or external third party

PROGRAMMING · 2.3

Data Structures

📋 List (2.3.1)

Purpose: Ordered, mutable collection that allows duplicate values. Access elements by zero-based index. Size can change during execution.

# Create
marks = [72, 85, 91, 68]

# Access by index (zero-based)
print(marks[0])       # 72 (first)
print(marks[-1])      # 68 (last)

# Modify
marks[2] = 95          # update item
marks.append(77)      # add to end
marks.remove(68)      # remove by value

# Iterate
for m in marks:
    print(m)

print(len(marks))      # length

When to use: Ordered collections where items may change — e.g. student names, queue of tasks, list of search results.

📐 Array (2.3.1)

Purpose: Fixed-size collection where all elements must be the same data type. More memory efficient than lists for large numerical datasets.

import array

# 'i' = signed integer type code
temps = array.array('i', [22, 19, 25, 17])

print(temps[1])        # 19
temps[0] = 23          # update allowed

# Common type codes:
# 'i' = int, 'f' = float, 'b' = signed byte

# Iterate
for t in temps:
    print(t)

When to use: Large collections of numerical data of the same type — sensor readings, pixel values, mathematical operations. More efficient memory use than a list.

🗂️ Dictionary (2.3.1)

Purpose: Unordered key-value pairs. Keys must be unique and immutable (strings, numbers). Values can be any type. Fast O(1) lookup by key.

# Create
student = {
    "name": "Ana",
    "grade": 85,
    "passed": True
}

# Access by key
print(student["name"])         # Ana
print(student.get("grade", 0)) # 85 (safe)

# Modify / Add
student["grade"] = 90
student["year"] = 2

# Check key exists
if "name" in student:
    print("Found")

# Iterate
for key, val in student.items():
    print(f"{key}: {val}")

When to use: Named attributes for a single entity — student records, configuration settings, frequency counts. Fast lookup when you know the key.

Choosing the Right Structure — Efficiency (2.9.3)

RequirementBest structureReason
Ordered items, may grow or shrinkListMutable, ordered, supports append/remove
Large dataset of same numerical typeArrayLess memory overhead than list; faster for maths
Named attributes for one entityDictionaryO(1) key lookup; human-readable keys
Need to know if a value exists quicklyDictionaryin operator is O(1) for dicts vs O(n) for lists

PROGRAMMING · 2.6

Sequence, Selection & Iteration

1️⃣ Sequence (2.6.1)

Instructions execute in the exact order they are written. No branching or repetition — the default mode of execution.

# Each line executes in order
x = 5
y = x * 2
z = y + 3
print(z)   # 13

2️⃣ Selection (2.6.2)

Choose which block to execute based on a condition. Python keywords: if, elif, else, match/case (3.10+).

# if / elif / else
if score >= 70:
    grade = "A"
elif score >= 50:
    grade = "C"
else:
    grade = "F"

# match/case (Python 3.10+)
match day:
    case "Mon": print("Start!")
    case "Fri": print("TGIF!")
    case _:     print("Midweek")

3️⃣ Iteration (2.6.3–2.6.5)

Repeat a block. Count-controlled: known number of repetitions. Condition-controlled: repeat until a condition becomes False.

# Count-controlled: for
for i in range(5):      # 0,1,2,3,4
    print(i)
for i in range(1,6):   # 1,2,3,4,5
    print(i)
for item in my_list:   # iterate list
    print(item)

# Condition-controlled: while
count = 0
while count < 5:
    count += 1
    print(count)

Loop Benefits & Drawbacks (2.6.5)

for Loop (count-controlled)while Loop (condition-controlled)
Use whenNumber of iterations is known in advance; iterating over a sequence/listNumber of iterations unknown; loop until a condition changes (e.g. user input validation)
BenefitsClean, readable; automatically manages the counter variable; less risk of infinite loopFlexible; can continue indefinitely until a complex condition is met
DrawbacksOff-by-one errors (range(n) gives 0 to n-1, not 1 to n); index must be tracked separately if neededRisk of infinite loop if condition never becomes False; condition must be carefully designed

PROGRAMMING · 2.8

Validation

Definition (2.8.1): Validation is an automatic check applied to data to ensure it is reasonable, sensible, complete and within acceptable boundaries — performed before the data is stored or processed. Validation prevents garbage data entering the system. Note: it does not verify that data is correct, only that it is acceptable.

✅ Presence Check

Ensures a mandatory field has not been left empty — that data has actually been entered.

def check_presence(value):
    if value == "":
        return "Field required"
    return "OK"

# Use for: username, password, name

📏 Length Check

Ensures input is within acceptable minimum and/or maximum character length limits.

def check_length(pw, min_l, max_l):
    n = len(pw)
    if n < min_l:
        return f"Min {min_l} chars"
    if n > max_l:
        return f"Max {max_l} chars"
    return "OK"

# Use for: password, username, postcode

🔢 Range Check

Ensures a value falls within an acceptable numerical range (inclusive or exclusive boundaries).

def check_range(age):
    if age < 0 or age > 120:
        return "Invalid age"
    return "OK"

# Use for: age, mark (0–100),
# temperature, quantity

🔤 Type Check

Ensures data is of the expected data type — prevents type errors during processing.

def check_type(value, expected):
    if not isinstance(value, expected):
        return f"Expected {expected}"
    return "OK"

check_type("hello", int)  # Fail
check_type(42, int)      # OK

Ensures data matches a required pattern — e.g. email, phone number, date, postcode.

🔑 Check Digit

An extra digit appended to a number, calculated from the other digits using a formula. Detects transcription errors (digit copied wrongly) and transposition errors (two adjacent digits swapped).

Examples: ISBN-13 book numbers, EAN barcodes, bank account sort codes, NHS numbers. The receiver re-calculates the check digit and compares it to the transmitted one — a mismatch means the data was corrupted in transit.

Characteristics of Robust Code (2.10.1)

  • Handles unexpected inputs — e.g. user enters "abc" in a numeric field; handles gracefully rather than crashing
  • Handles unexpected terminations — e.g. file not found, network connection drops, division by zero; catches exceptions and recovers
  • Produces specific and meaningful error messages — tells the user what went wrong, how to fix it, and what input is expected
# Robust input validation loop

        age = int(input("Enter age (0-120): "))

    except ValueError:
        print("Please enter a whole number.")

# Robust file reading

    with open("data.txt") as f:
        data = f.read()
except FileNotFoundError:
    print("Error: data.txt not found.")

Debugging Process (2.10.2–2.10.5)

Debugging: The process of locating and correcting errors in code. Essential for producing robust, reliable solutions.

  1. Reproduce the error — identify the exact input or conditions that cause the problem to occur consistently
  2. Locate the error (2.10.4) — use print statements, an IDE debugger (breakpoints, step-through), or construct a trace table by hand
  3. Understand the error — determine exactly why this line/condition produces the wrong result or exception
  4. Fix the error (2.10.5) — correct the code with the minimum necessary change
  5. Test the fix — run the original failing test case, plus all other tests, to confirm the fix works and hasn't introduced new bugs

Types of Errors

TypeCauseWhen detected

LogicProgram runs without errors but produces wrong output (wrong formula, wrong condition)Testing — requires human analysis to find
💡 Role of debugging in robustness (2.10.3): Every round of debugging makes code more robust. The goal is to anticipate all possible failure conditions and handle them gracefully, so no user input or external event can cause an unhandled crash.

Searching Algorithms

🔍 Linear Search (2.11.1)

How it works: Start at the first element and check each element one by one from left to right until the target is found or the entire list has been checked.

def linear_search(lst, target):
    for i in range(len(lst)):
        if lst[i] == target:
            return i        # return index
    return -1              # not found

# Example trace: find 7 in [3, 7, 2, 9]
# i=0: 3 == 7? No
# i=1: 7 == 7? Yes → return 1
CaseComparisonsWhen it occurs
Best1Target is the first element
Averagen/2Target is in the middle
WorstnTarget is last or not present
Benefits: Works on any list — unsorted or sorted. Simple to implement and understand. No extra memory needed. Drawbacks: Very slow for large datasets — must check every element in worst case.

⚡ Binary Search (2.11.1)

How it works: Requires a sorted list. Compare the target to the middle element. If target is smaller, discard the upper half; if larger, discard the lower half. Repeat with the remaining half until found or search space is empty.

def binary_search(lst, target):
    low, high = 0, len(lst) - 1
    while low <= high:
        mid = (low + high) // 2
        if lst[mid] == target:
            return mid      # found
        elif lst[mid] < target:
            low = mid + 1   # search right half
        else:
            high = mid - 1  # search left half
    return -1

# Trace: find 7 in [1,3,5,7,9,11,13]
# low=0,high=6 → mid=3, lst[3]=7 ✓ return 3
CaseComparisonsWhen it occurs
Best1Target is the middle element
Averagelog₂(n)Target found after several halvings
Worstlog₂(n)Target not present — search space reduced to zero
Benefits: Extremely fast for large datasets — 1,000,000 items needs at most 20 comparisons. Drawbacks: List must be sorted first (sorting has its own cost); more complex to implement correctly.

Sorting Algorithms

🫧 Bubble Sort (2.11.2)

How it works: Make repeated passes through the list. On each pass, compare adjacent pairs and swap them if they are in the wrong order. Largest values gradually "bubble" to the end. Stop when a full pass is made with no swaps.

def bubble_sort(lst):
    n = len(lst)
    for i in range(n-1):
        swapped = False
        for j in range(n-i-1):
            if lst[j] > lst[j+1]:
                lst[j],lst[j+1]=lst[j+1],lst[j]
                swapped = True
        if not swapped: break
CasePerformance
BestO(n) — already sorted
WorstO(n²) — reverse sorted
AverageO(n²)
MemoryO(1) — in-place

✅ Simple; easy to implement; O(n) for nearly-sorted data with optimisation

❌ Very slow for large unsorted lists; many unnecessary comparisons

📥 Insertion Sort (2.11.2)

How it works: Build a sorted sub-list one element at a time. Take the next unsorted element and insert it into the correct position in the sorted portion by shifting elements right.

def insertion_sort(lst):
    for i in range(1, len(lst)):
        key = lst[i]
        j = i - 1
        while j >= 0 and lst[j] > key:
            lst[j+1] = lst[j]
            j -= 1
        lst[j+1] = key
    return lst
CasePerformance
BestO(n) — already/nearly sorted
WorstO(n²) — reverse sorted
AverageO(n²)
MemoryO(1) — in-place

✅ Excellent for small or nearly-sorted data; stable sort; simple

❌ Slow for large, randomly ordered datasets

🔀 Merge Sort (2.11.2)

How it works: Divide-and-conquer. Recursively split the list in half until each sub-list has one element, then merge pairs of sub-lists back together in sorted order.

def merge_sort(lst):
    if len(lst) <= 1:
        return lst
    mid = len(lst) // 2
    left = merge_sort(lst[:mid])
    right = merge_sort(lst[mid:])
    return merge(left, right)

Acceptable Use Policy (AUP) (4.2.4)

Purpose: An AUP is a formal document that defines the rules and restrictions governing how users are permitted to use an organisation's digital systems, network, internet connection and data.

Typical Content of an AUP

SectionWhat it coversExample clauses
Permitted activitiesWhat users are allowed to do with company systemsAccess work email; use approved software; store work files on company drive
Prohibited activitiesActions that are expressly bannedInstalling unauthorised software; accessing inappropriate websites; sharing login credentials; downloading pirated content
Working practicesRequirements around confidentiality and data handlingLock screen when leaving workstation; use encrypted email for sensitive data; don't discuss client data in public
Communication etiquetteStandards for professional communicationsRepresent the organisation professionally in all emails; no offensive language; include appropriate disclaimers
Sanctions and penaltiesConsequences of violating the AUPVerbal warning → written warning → dismissal → legal action depending on severity; potential criminal prosecution under CMA 1990
