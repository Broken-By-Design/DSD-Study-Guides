PROBLEM SOLVING · 1.1

Computational Thinking

🧠 What is Computational Thinking?

A problem-solving approach that involves breaking down complex problems into manageable steps that a computer can solve. It is a skill used by both humans and machines and forms the foundation of all programming.

🔎 Key Components (1.1.4–1.1.5)

  • Decomposition — breaking a problem into smaller sub-problems
  • Pattern recognition — identifying similarities and repeated patterns
  • Abstraction — removing unnecessary detail to focus on what matters
  • Algorithm design — creating a step-by-step solution

🌟 Why It Matters (1.1.1–1.1.3)

  • Allows complex real-world problems to be modelled and solved
  • Helps programmers write clear, efficient, maintainable code
  • Underpins all areas of computer science and software development
  • Used in AI, data science, networking and systems design

Selecting the Right Component (1.1.17)

ComponentWhen to use itExample
DecompositionProblem is large, complex or multi-partSplit an e-commerce site into login, basket, payment modules
Pattern RecognitionSimilar sub-problems appear repeatedlySorting multiple lists uses the same algorithm each time
AbstractionToo much detail obscures the key issueA map shows roads but not individual blades of grass
Algorithm DesignA repeatable, precise solution is neededWriting pseudocode before coding
Exam tip: Questions often ask you to apply a specific component to a given scenario. Name the component, define it, then explain how it applies. Always use the context given.
PROBLEM SOLVING · 1.1

Decomposition, Patterns & Abstraction

🤛 Decomposition (1.1.6–1.1.10)

  • Breaking a large problem into smaller, more manageable sub-problems
  • Each sub-problem can be solved independently and tested separately
  • Enables teams to work in parallel on different modules
  • Sub-problems can themselves be decomposed further (recursively)
  • Example: a library system = search books + borrow books + return books + manage members

📊 Pattern Recognition (1.1.11–1.1.12)

  • Identifying similarities, trends and repetitions in data or problems
  • Reuse solutions that have already worked for similar problems
  • Helps generalise algorithms so they work on multiple inputs
  • Example: every user login needs: validate username → check password → grant/deny access
  • Used in ML/AI to identify features in data

🌊 Abstraction (1.1.13–1.1.16)

  • Filtering out unnecessary detail to focus only on what is relevant
  • Hides complexity from the user or programmer
  • Makes complex systems easier to understand and build
  • Example: a map shows roads and landmarks but not soil composition
  • Object-Oriented Programming uses abstraction through classes

Practical Examples of Abstraction

📱 Drawing a Cat (Decomposed)

  • 1. Draw body → 2. Draw head → 3. Draw legs
  • 4. Draw tail → 5. Draw ears → 6. Draw whiskers → 7. Colour in
  • Each step is a sub-task that can be tackled individually
  • Generalising to ‘draw any animal’ is further abstraction

🚛 Real-World Abstractions

  • GPS navigation: shows roads, not buildings’ internal layouts
  • Online shop: hides database queries behind a ‘Buy Now’ button
  • ATM: user sees ‘Withdraw Cash’ — not the bank’s internal systems
  • Sorting algorithm: hides comparisons — user just calls sort()
PROBLEM SOLVING · 1.2

Algorithms & Flowcharts

📄 What is an Algorithm? (1.2.1)

  • A precise, step-by-step set of instructions to solve a problem
  • Must be unambiguous — each step must have only one interpretation
  • Must be finite — it must eventually reach a conclusion
  • Must be correct for all valid inputs, not just some
  • Can be expressed as written description, pseudocode or flowchart

📋 Which Method? (1.2.2)

  • Written description — easy to understand for non-programmers, but can be ambiguous
  • Pseudocode — structured like code but language-independent; precise and unambiguous
  • Flowchart — visual, shows flow of control clearly; good for decisions and loops
  • Often used together: pseudocode for logic; flowchart for structure

Flowchart Symbols

SymbolNamePurpose
◯Terminal (Oval)START or END of the algorithm
▱Input/Output (Parallelogram)Data entering or leaving the system
▭Process (Rectangle)A calculation or assignment, e.g. x = x + 4
◇Decision (Diamond)A yes/no question that branches the flow
▭▭Sub-process (Double-sided rectangle)A sub-task or function call
◯Connector (Circle)Links different parts of a large flowchart
→ArrowShows the direction of flow through the chart
Bus pass example: Age input → Is age < 16? (Yes → “Free child pass”) → Is age ≥ 65? (Yes → “Free OAP pass”) → Otherwise → “Cost is £10”. Decision diamonds map directly to IF statements in code.
PROBLEM SOLVING · 1.2

Pseudocode: Selection & Iteration

🔌 Sequence

The computer executes instructions one after another in the order given. Order matters — incorrect ordering can cause wrong results or errors.

GET name FROM KEYBOARD
OUTPUT "Hello " + name

🤔 Selection (IF / CASE)

  • IF…THEN…END IF — runs block only if condition is true
  • IF…THEN…ELSE…END IF — two possible paths
  • CASE / SWITCH — multiple branches based on one variable’s value
IF age >= 18 THEN
  OUTPUT "Adult"
ELSE
  OUTPUT "Minor"
END IF

Iteration (Loops)

🔄 FOR Loop

Runs a fixed number of times. The counter variable increments each iteration.

FOR i = 1 TO 5
  OUTPUT i
END FOR

🔄 WHILE Loop

Pre-condition checked before each iteration. May never run if condition starts false.

WHILE count < 10
  SET count = count + 1
END WHILE

🔄 REPEAT…UNTIL Loop

Post-condition checked after each iteration. Always runs at least once.

REPEAT
  GET input FROM KEYBOARD
UNTIL input > 0
Key differences: FOR = known count; WHILE = condition checked first (may not run); REPEAT…UNTIL = runs at least once. Iteration can also be achieved through recursion (a function calling itself).
PROBLEM SOLVING · 1.3

Testing, Debugging & Root Cause Analysis

🔎 Dry Run Testing

Manually tracing through an algorithm with test data, recording variable values at each step in a trace table. Identifies logic errors without running code.

🐞 Types of Errors

  • Syntax error — code breaks grammar rules; stops it running
  • Logic error — code runs but gives wrong output
  • Runtime error — crash during execution (e.g. dividing by zero)

🌟 Root Cause Analysis (1.3.4)

  • Find the origin of a problem, not just treat its symptoms
  • Five Whys technique — ask ‘why?’ five times to reach the root
  • Credited to Sakichi Toyoda (founder of Toyota Industries)
  • Prevents the same issue recurring

Five Whys Example

Why?Answer
1. Why did the site crash?The database stopped responding
2. Why did the database stop?It ran out of storage space
3. Why did it run out of space?Log files were never deleted
4. Why were they never deleted?No automated cleanup was set up
5. Why was no cleanup set up?Root cause: the deployment checklist didn’t include storage management

🆕 When to Use Root Cause Analysis

  • When a problem is difficult or recurring
  • When consequences are serious (safety-critical systems)
  • When a ‘sticking plaster’ fix keeps failing

🛠️ Subsequent Actions

  • Review the whole system for related issues
  • Rework database structure if needed
  • Update processes for future projects (e.g. add unit testing from the start)

🔑 Constants (2.2.2)

  • Values that do not change during program execution
  • Set once at the start; cannot be reassigned
  • Python convention: ALL_CAPITALS for constant names (PI = 3.14159)
  • Benefit: only one place to edit if the value ever changes
  • Examples: PI, VAT_RATE = 0.2, MIN_AGE = 18

💾 Variables (2.2.1)

  • Named storage locations whose values can change during execution
  • Assigned using = in Python
  • Examples: counters, user inputs, running totals
  • Local variable — only accessible inside the function it’s declared in
  • Global variable — accessible throughout the entire program

Type Conversion Functions (2.2.3)

int(x) — to integer float(x) — to decimal str(x) — to string bool(x) — to boolean
Scope (2.2.4–2.2.5): Prefer local variables where possible — they prevent naming conflicts between team members, reduce bugs, and keep functions self-contained. Global variables risk being accidentally modified from anywhere in the code.
PROGRAMMING · 2.3–2.4

Data Structures & Operators

📋 List (Array) (2.3.1)

  • Ordered, indexed collection of items
  • In Python: names = ["Noah", "Leo", "Theo"]
  • Access by index (0-based): names[0] = "Noah"
  • Can add/remove items; mutable
  • Iterate with a FOR loop or for item in list

📜 Dictionary (2.3.1)

  • Stores data as key–value pairs
  • capitals = {"France": "Paris", "Japan": "Tokyo"}
  • Access: capitals["France"] returns "Paris"
  • Keys must be unique; values can be any type
  • Useful for lookup tables and records

🛠️ Tuple / Set

  • Tuple — ordered and immutable; values cannot change after creation
  • Set — unordered, no duplicates; good for membership testing
  • Both are less commonly used than lists and dicts in T Level scenarios

PROGRAMMING · 2.6

Sequence, Selection & Iteration in Code

🔄 FOR Loops in Python

  • for i in range(0, 5): — runs 5 times (0,1,2,3,4)
  • for item in my_list: — iterates through each element
  • Use a FOR loop when the number of iterations is known
for i in range(1, 6):
  print(i, ":", name)

🔄 WHILE Loops in Python

  • Condition checked before each iteration
  • May never run if condition is initially False
  • Use when the exit condition depends on user input or computed value
while count < 10:
  count = count + 1

Selection in Python (2.6.2)

IF / ELIF / ELSE

if score >= 70:
  print("Pass")
elif score >= 50:
  print("Merit")
else:
  print("Fail")

Each elif is only checked if all previous conditions were False.

Match / Case (Python 3.10+)

match subject:
  case 'Computing':
    print("Me too!")
  case 'Art':
    print("Nice!")
  case _:
    print("Interesting")

Cleaner alternative to long if/elif chains. case _ is the default.

Benefits & Drawbacks of Loops (2.6.5)

TypeBenefitDrawback
FORClear, predictable; easy to readCannot easily exit early without break
WHILEFlexible; runs until any condition is metRisk of infinite loop if condition never becomes False
RecursionElegant for problems defined in terms of themselvesRisk of stack overflow; harder to debug
PROGRAMMING · 2.7–2.8

Functions, Procedures & Validation

🔨 Functions (2.7.1–2.7.2)

  • A named, reusable block of code that performs a specific task
  • Function — returns a value back to the caller
  • Procedure — performs an action but does NOT return a value
  • Parameters pass values in; the return statement sends a result back
  • Reduces repetition (DRY — Don’t Repeat Yourself)
  • Makes code modular, readable and easier to test

📄 Pre-written Functions (2.7.3–2.7.4)

  • len(x) — length of a string or list
  • input() — gets user input as a string
  • print() — outputs to console
  • range(start, stop) — generates a sequence of integers
  • round(x, dp) — rounds to specified decimal places
  • Library functions: math.sqrt(), random.randint()

Input Validation (2.8.1)

🚫 Presence Check

Ensures a field is not left empty. E.g. name cannot be blank on a registration form.

🔢 Range Check

Ensures a value falls within acceptable limits. E.g. age must be between 0 and 120.

🔌 Type Check

Confirms the data is the correct data type. E.g. a phone number must contain only digits.

🌏 Format Check

Data matches a specific pattern. E.g. a date must be in DD/MM/YYYY format; email must contain @.

📋 Length Check

Value length is within limits. E.g. password must be 8–20 characters long.

👤 Lookup / List Check

Value must match one from a predefined set. E.g. gender must be M, F or X from a dropdown.

Validation vs Verification: Validation checks the data meets rules (format, range, type). Verification confirms the data matches its original source — e.g. typing a password twice (double-entry) or visually checking a form against a paper document.
PROGRAMMING · 2.9–2.10

Code Quality & Robust Code

🔎 Naming Conventions (2.9.5)

  • camelCase — myVariableName (Java, JavaScript)
  • snake_case — my_variable_name (Python)
  • PascalCase — MyClassName (classes in most languages)
  • Constants in ALL_CAPS: MAX_SPEED = 100
  • Names should be descriptive, not x or temp1

📄 Code Readability (2.9.6)

  • Use meaningful variable and function names
  • Add comments to explain complex sections
  • Consistent indentation (Python requires this)
  • Avoid overly long functions — break into smaller ones
  • Group related code together; use blank lines to separate sections

🐞 Types of Errors (2.9.1–2.9.4)

  • Syntax — breaks language rules; won’t compile/run
  • Logic — code runs but produces wrong output
  • Runtime — crashes during execution
  • Semantic — code does something other than intended

🆕 What is Robust Code? (2.10.1)

  • Code that handles unexpected inputs gracefully, without crashing
  • Uses validation to reject bad inputs before they cause problems
  • Uses exception handling (try/except in Python) to catch runtime errors
  • Provides meaningful error messages to the user
  • Continues to work correctly under edge cases and boundary conditions

🛠️ Debugging Techniques (2.10.2–2.10.3)

  • Print statements — output variable values at key points
  • IDE debugger — set breakpoints; step through code line by line
  • Trace tables — manually trace variables through an algorithm
  • Rubber duck debugging — explain code aloud to find logic errors
  • Unit testing — test individual functions in isolation
PROGRAMMING · 2.11–2.12

Common Algorithms & Testing

Searching Algorithms (2.11.1, 2.11.3)

🔎 Linear Search — O(n)

  • Checks every element in order from start to end
  • Works on unsorted data
  • Worst case: check every element (n comparisons)
  • Simple to implement; good for small datasets

🔍 Binary Search — O(log n)

  • Only works on sorted data
  • Compares target to middle element; discards half each time
  • Much faster than linear for large datasets
  • E.g. searching 1,000 items takes at most 10 comparisons

Sorting Algorithms (2.11.2–2.11.3)

AlgorithmComplexityBest forDrawback
Bubble SortO(n²)Small or nearly-sorted datasetsVery slow on large datasets
Insertion SortO(n²)Small datasets; efficient if almost sortedSlower than merge sort on large data
Merge SortO(n log n)Large datasets; reliable performanceMore complex to implement

Testing Methods (2.12.2)

🟢 White Box (Open Box)

Tester has full knowledge of the internal code. Can test every path, branch and boundary. Removes bias from unknown internals.

🔴 Black Box (Closed Box)

Tester has no knowledge of internal code. Tests only input vs expected output. No bias from knowing how it works.

🔁 Regression Testing

After a bug fix or new feature, tests all previously-passing tests again to ensure nothing new was broken by the change.

📈 Performance / Stress Testing

Tests under high load, endurance, spike traffic and maximum data volume. Checks speed, response time and reliability under pressure.

👤 Acceptance Testing

User verifies the system meets their requirements. They should explore freely, not be given a guided walk-through that hides problems.

📄 Test Plans & Test Data (2.12.4)

  • Normal — valid, typical inputs
  • Boundary — values at the edge of acceptable range
  • Erroneous — invalid inputs that should be rejected

EMERGING ISSUES · 3.1

Impact of Digital Technologies

🏢 Impacts on Organisational Culture (3.1.1)

  • New working practices: hybrid/remote working, flexible hours, hot-desking
  • Communication: shift from face-to-face to email, Slack, Teams, video calls
  • Productivity: 24/7 availability expectations; employees reachable at all times
  • Staff monitoring: CCTV, keystroke logging, website monitoring — raises privacy concerns
  • Automation: AI chatbots, robotic process automation, reducing manual roles
  • Skill shifts: jobs require coding, data analysis and digital tools increasingly

🌎 Impacts on Society (3.1.1)

  • Digital footprint: data trail from browsing, social media, purchases
  • Changing behaviours: reduced face-to-face social skills; increased screen time affecting sleep
  • Digital identities: curated social media personas may harm self-esteem
  • Surveillance: CCTV with facial recognition raises human rights concerns
  • Globalisation: trends, products and information spread worldwide within minutes
  • Social networks: wider personal and professional connections globally
Professional Development (3.1.4): Increased industry competence, improved employability and employment security, access to up-to-date industry standards, and adherence to codes of best practice all result from ongoing digital learning and professional development.
EMERGING ISSUES · 3.1

Digital Inclusion & End-User Characteristics

🍀 Digital Inclusion (3.1.2)

  • Ensuring everyone can access and benefit from digital technologies
  • Suitable hardware/software: affordable devices, assistive tech
  • Connectivity: rural broadband gaps; 5G rollout
  • Checking dataset bias: AI trained on biased data disadvantages minorities
  • Accessibility regulations: WCAG guidelines; Equality Act 2010
  • Codes of best practice: designing for all users from the outset

👤 End-User Characteristics (3.1.3)

  • Age: older users may struggle with fine motor control or small text; younger users grew up digital
  • IT skills: interfaces must suit novice and expert users
  • Literacy levels: language complexity should match audience
  • Internal vs external: employee-facing vs customer-facing platforms differ
  • Cultural differences: colours, symbols and phrases differ across cultures
  • Additional needs: screen readers, high contrast, voice recognition support

AI Bias (3.1.2)

🚫 Why AI Bias Occurs

  • AI systems are trained on historical datasets that may reflect past societal biases
  • Models predict based on the most common patterns — disadvantaging minorities
  • Example: hiring algorithms rejecting CVs from underrepresented groups
  • Example: facial recognition working less accurately on darker skin tones

✅ Addressing AI Bias

  • Anti-bias training data — representative and diverse datasets
  • Human oversight of automated decisions
  • Explainable AI (XAI) — understanding why a decision was made
  • Regular audits of AI outputs for discriminatory patterns
EMERGING ISSUES · 3.2

IoT, AI & Machine Learning

🌐 Internet of Things (IoT) (3.2.1)

  • Everyday objects with embedded sensors connected to the Internet
  • Examples: smart meters, thermostats, doorbells, wearables, factory sensors
  • Edge computing: processing data locally on the device rather than sending to the cloud — reduces latency
  • Risks: weak default passwords, surveillance concerns, data privacy, hacking (baby monitors, kettles)
  • Regulation: unique passwords required; providers must release security updates

🧠 Artificial Intelligence (AI) (3.2.1)

  • Systems that perform tasks requiring human-like intelligence
  • Current uses: chatbots, fraud detection, medical imaging, recommendation systems, facial recognition
  • Generative AI: ChatGPT (text), DALL·E (images), video/audio generation, deepfakes
  • Machine Learning: learns patterns from data; improves over time without explicit programming
  • Deep Learning: uses neural networks modelled on the human brain; powers image/speech recognition

EMERGING ISSUES · 3.2

Blockchain, Environment & Emerging Tech

🔗 Blockchain (3.2.1)

  • A decentralised, distributed digital ledger — data stored in linked cryptographic blocks
  • Very secure: no central server; multiple copies across many nodes
  • Uses: cryptocurrency (Bitcoin, Ethereum), smart contracts, supply chain tracking, voting systems, healthcare records, digital identity
  • Drawbacks: very high energy use, slower than traditional databases, legal uncertainty, hard to integrate with existing systems

🌿 Environmental Impacts (3.2.1)

  • Always-on devices (routers, TVs on standby, voice speakers) consume constant energy
  • Mining of rare earth metals damages local environments and communities
  • WEEE Regulations — suppliers must responsibly take back and dispose of electronic products at end of life
  • Data centres have enormous power and cooling demands
  • Growing pressure on organisations to reduce carbon footprint and source ethically mined materials

🚀 Extended Reality (3.2.1)

  • AR (Augmented Reality) — overlays digital content on the real world (e.g. Snapchat filters, warehouse picking systems)
  • VR (Virtual Reality) — fully immersive digital environment (gaming, training simulations, therapy)
  • The Metaverse — persistent shared virtual environment; still emerging

🛠️ Open Source Software

  • Source code freely available to view, modify and distribute
  • Examples: Linux, Python, Firefox, LibreOffice
  • Creative Commons licences allow sharing with varying restrictions
  • Reduces costs; promotes collaboration; community-maintained security patches

🤖 Autonomous Machines

  • Self-driving vehicles, delivery drones, humanoid robots, factory equipment
  • AI agents that plan, execute and monitor tasks with minimal human input
  • Raises ethical questions: accountability in accidents; job displacement
  • Requires advances in sensors, ML and real-time edge computing

LEGISLATION · 4.1

Health & Safety + Data Protection

⚠️ Health & Safety Legislation (4.1.1–4.1.2)

  • Health & Safety at Work Act 1974 — employers must provide a safe working environment
  • Display Screen Equipment (DSE) Regulations — applies to anyone using screens for 1+ continuous hours
  • Risks: eye strain, headaches, back/neck/shoulder pain, RSI (repetitive strain injury), stress
  • Mitigations: ergonomic keyboards/mice, adjustable lighting, footrests, telephone headsets, speech recognition, regular breaks, correct chair height
  • Also applies to home workers during COVID and beyond

🔒 UK GDPR & Data Protection Act 2018 (4.1.3)

  • Replaced earlier Data Protection Acts (1984, 1998)
  • Data controller — organisation that holds and decides how to use personal data
  • Data subject — individual whose data is held
  • ICO — Information Commissioner’s Office; enforces GDPR in the UK
  • Fines: up to £17.5m or 4% of global annual turnover (whichever is larger)

The 8 Principles of UK GDPR

1Lawful, fair & transparentData must be collected and used fairly with clear justification
2Purpose limitationData only used for the specific stated purpose

4AccuracyData must be accurate and kept up to date

6Integrity & confidentialityKept secure against unauthorised access or loss

8International transfersThird parties must uphold the same standards
LEGISLATION · 4.1

Computer Misuse, Equality & IP Law

🔐 Computer Misuse Act 1990 (4.1.4)

Three criminal offences were created:

  • Offence 1: Unauthorised access to programs or data (must prove they knew they were unauthorised)
  • Offence 2: Unauthorised access with intent to commit a further offence
  • Offence 3: Unauthorised modification of programs or data (includes deliberately introducing a virus)

Updated by Police & Justice Act 2006 and Serious Crime Act 2015 (added distributing malware). UK courts have jurisdiction where a ‘significant link’ to Britain is demonstrated.

⚖️ Equality Act 2010 (4.1.5)

  • Protects against discrimination on the basis of protected characteristics
  • Age, disability, gender reassignment, marriage & civil partnership, pregnancy & maternity, race, religion/belief, sex, sexual orientation
  • Applies to digital products — exclusionary design can breach the Act
  • Combined with WCAG (Web Content Accessibility Guidelines) for web/app accessibility

Intellectual Property (IP) Legislation (4.1.6)

📄 Copyright

  • Automatic protection for original creative works (code, text, music, images)
  • Code belongs to the developer; if created at work it likely belongs to the employer
  • No registration required; lasts life + 70 years typically

📜 Patents

  • Protects inventions for up to 20 years
  • Must be: new, inventive (non-obvious), and capable of industrial application
  • Expensive (£1000s) and slow to obtain; must be renewed annually after year 4
  • Gives exclusive rights to use the invention

🏗️ Registered Designs

  • Protects the appearance of a product (shape, colour, texture)
  • Registered: 25 years (renewed every 5); can sue without proving deliberate copying
  • Unregistered: 15 years (UK); must prove deliberate duplication
  • IP Act 2014 introduced criminal penalties (up to 10 years) for copying registered designs
LEGISLATION · 4.2

Codes of Conduct, AUPs & Whistleblowing

📚 Codes of Conduct (4.2.1–4.2.2)

  • Professional bodies set standards that guide ethical behaviour in the digital industry
  • BCS — British Computer Society; covers public interest, professional competence, duty to authority and profession
  • CIISec — Chartered Institute of Information Security
  • ACM — Association for Computing Machinery (international code of ethics)
  • All codes stress: confidentiality, integrity, competence, public interest, and honesty

📋 Acceptable Use Policies (4.2.4)

  • A document that defines what is and is not permitted when using an organisation’s IT systems
  • Required by employers to reduce risk of legal liability (links to Computer Misuse Act — users must know what is unauthorised)
  • Also used by websites and apps (terms & conditions users accept on sign-up)
  • Covers: data access, personal use, social media, password sharing, device use, and remote working
  • Employees must be trained on it — ‘I didn’t know’ is not a defence

📢 Digital Industry Standards (4.2.3)

  • ISO/IEC 27001 — international standard for information security management
  • WCAG — web accessibility guidelines (levels A, AA, AAA)
  • Agile / Scrum / ITIL — project management and service delivery standards
  • Standards give consistent quality, build trust, and ensure legal compliance

🗣️ Whistleblowing (4.2.5)

  • Reporting wrongdoing, illegal activity or health & safety concerns through proper channels
  • Protected by Public Interest Disclosure Act 1998 — workers cannot be unfairly dismissed for whistleblowing
  • Digital workers may uncover unethical data use, security breaches or IP theft
  • Should be reported to line managers, an ethics committee or regulatory bodies (e.g. ICO)
Relationship between software development and guidelines (4.2.6): Developers must consider GDPR (data handling), Equality Act (accessible design), Computer Misuse Act (secure systems), copyright (using open-source correctly), and their professional code of conduct at every stage of the software development lifecycle.
