TASK 1 · PO1

Task 1 — Analysing a Problem & Designing a Solution

Task 1 primarily assesses PO1 (Requirements Analysis — 24 marks) but also contributes to PO2, PO4, PO5 and PO6. This is where you show you understand the problem fully before writing any code. p.59–64

Activity A — Research & Requirements

  • Explore the client request / project brief
  • Research the specific context and market environment
  • Identify functional requirements — what the system must do: inputs, data, processing logic, deployment platforms
  • Identify non-functional requirements — how the system performs: security, accessibility, scalability, responsiveness, load handling, reliability
  • Define key performance indicators (KPIs)
  • Create user acceptance criteria
  • Identify risks and mitigations
  • Schedule tasks, milestones and subtasks
  • Estimate costs and allocate resources

Activity B — User Analysis

Select and use business analysis models to aid problem solving:

  • User stories — As a [user], I want [feature] so that [benefit]
  • Activity diagrams — show the flow of actions
  • Mind maps — brainstorm requirements
  • Product road maps — planned feature timeline
  • Process diagrams — step-by-step workflows
  • Entity relationship diagrams — data relationships

Apply computational thinking: decomposition, pattern recognition, abstraction to split the problem into discrete objects.

Activity C — Design & AI Use

  • Create interface designs (wireframes, UI layouts)
  • Plan algorithms to solve key problems (flowcharts, pseudocode)
  • Create data requirements designs (ERDs, data dictionaries)
  • Use generative AI tools to produce short snippets of code for specific functions — must be reviewed and refined
  • Document your AI prompts and the outputs you received
  • Justify how AI output was adapted to meet requirements
Design documents must be detailed enough for a third party to build the solution from them.
💡 High-mark strategy: Don't just list requirements — justify them. For each functional and non-functional requirement, explain why it matters in this context. For risks, don't just name them — assess likelihood vs seriousness and state specific mitigations. Use the correct technical vocabulary throughout.

CONTENT AREA 1.3

Project Methodologies

Select and justify an appropriate methodology for your project. You must understand the key features of each and explain why your choice suits the brief. p.61–63

Agile

  • Incremental delivery: sprint → epic → story → spikes
  • Each sprint delivers additional functionality
  • Requirements can be altered throughout
  • Clients see working products at each iteration
  • Can lack formal documentation
  • Cost-effective; cancelled projects still produce usable code

Scaled Agile

  • Expanded increments: sprint, epic, story, spikes, product increments
  • Final iteration focuses on stability
  • Provides cohesive reporting across multiple teams
  • Suitable for larger organisations

Waterfall

  • Rigidly structured, systematic steps — each phase completes before the next begins
  • Progress measured by artefacts completed
  • High initial costs; slower return on investment
  • High importance on documentation
  • Limited user interaction
  • Cancelled early → no usable product

Rapid Application Development (RAD)

  • Focus on quick prototypes systematically improved
  • Emphasis on developing all features first, quality second
  • Relies heavily on reuse of previously developed code
  • Suitable for small-to-medium projects

Lean

  • Emphasis on eliminating waste: unnecessary features, repeated tasks, complex solutions, ineffective communication, unnecessary changes
  • Decisions made last minute to reduce waste
  • Short iteration cycles; fast delivery of working versions
  • Suitable for small teams with limited resources

User Centred Design (UCD)

Characteristics: empathetic, iterative, interdisciplinary

Iterative stages: understand context → specify user requirements → design solution → assess against requirements → repeat

Key questions: Who is the user? What do they want to achieve? How/when do they interact? Why is the product used? What is the UX?

TASK 2 · PO2

Task 2 — Developing the Solution

Task 2 is the heart of PO2 (57 marks — 39.6% of the total). Quality code, strong design, thorough testing and clear documentation are all assessed here. Internet access is permitted. p.68–78

What You Must Produce

  • A working software solution using at least two appropriate languages
  • Front-end and back-end components
  • A user interface applying UX/UI design principles
  • Connection to data sources (database/API)
  • Evidence of appropriate testing at multiple stages
  • Commented, well-structured code following coding conventions
  • Prototype created and iteratively refined
  • Version control used throughout

UX Design Principles to Apply p.70

  • Consistency — intuitive, aesthetically pleasing, promotes brand recognition
  • Information hierarchy — navigate the product more easily
  • Visual hierarchy — important content stands out
  • Confirmation — user is aware of any actions performed
  • User control — navigation, workflow, error correction
  • Accessibility — appropriate for a wide range of users

Coding Conventions p.74

  • Naming conventions — meaningful, camelCase or snake_case
  • Code annotations/commenting — explain logic, not just what
  • Modularisation — separate functions/classes by responsibility
  • Structure/indentation — consistent, clear visual layout
  • Version control — commit regularly with meaningful messages
  • Avoid deeply nested loops and copy-pasted procedural code
  • Good exception handling and defensive programming
  • Minimal use of global variables; prefer local scope
  • Use of constants for fixed values

12 Factor Principles (Good Practice) p.74

  • One codebase tracked in revision control, many deploys
  • Explicitly declare and isolate dependencies
  • Store config in the environment
  • Treat backing services as attached resources
  • Strictly separate build and run stages
  • Execute app as one or more stateless processes
  • Treat logs as event streams
CONTENT AREA 4

Design Approaches

You must be able to select and justify appropriate design approaches and apply them within your chosen project methodology. Each has different implications for how you structure and build your solution. p.68–69

Function Orientated (Top-Down)

  • Consideration of how data flows through the system
  • System made up of many sub-systems (functions)
  • Use of data flow diagrams to show how each function handles/changes data
  • Problem is logically broken down (divide and conquer) based on what each function should do

Object-Oriented (OOP)

  • Core concept: making code reusable through methods, classes, objects/instances
  • Characteristics: encapsulation, data abstraction, polymorphism, inheritance
  • Common OOP design patterns: creational, structural, behavioural

Data Model Design

  • Visualisation of the data needed and how it will be organised
  • Common models: conceptual, logical, physical, hierarchical, relational
  • Tools: Entity Relationship Model (ERD), Unified Modelling Language (UML)
  • Normalisation to third normal form

Test-Driven Development (TDD)

  • Each new feature starts by writing a test first that defines the required function
  • Steps: Add test → Run tests (new fails) → Write code → Run & refactor → Repeat
  • Ensures every piece of code is covered by a test

Behaviour-Driven Development (BDD)

  • Development is informed by the required behaviour of a software unit
  • Desired behaviours must have business value and relate to specified requirements
  • Specification structure: title, narrative, acceptance criteria
  • Written in plain language stakeholders can understand

Functional Design

  • Program built as a set series of modules each performing a single defined process/function
  • Characteristics: recursion, closures, first-class functions, higher-order functions, anonymous functions, currying
  • Components: arguments, statements, blocks, procedures, functions/sub-routines

Connecting Code to Data Sources

Your solution must connect to data sources. Know the different connection methods, how to extract/store/update/delete data, and how to justify your chosen approach. p.75–76

API (Application Programming Interface)

  • Types of requests and request methods (GET, POST, PUT, DELETE)
  • Endpoints — specific URLs for accessing resources
  • Retrieve and parse data (JSON/XML responses)
  • Display data in the UI
  • API keys — authentication for access

JDBC (Java Database Connectivity)

  • Core API for Java/SQL connections
  • Driver manager — manages database drivers
  • Connection statement — establishes the link
  • Prepared statement — parameterised queries (prevents SQL injection)
  • Result set — holds query results
  • SQL queries: SELECT, INSERT, UPDATE, DELETE

ODBC (Open DB Connectivity)

  • Application — the software making the request
  • Driver manager — routes requests
  • Driver — translates to database-specific calls
  • Data source — the actual database
  • More vendor-neutral than JDBC

Content Management Systems (CMS) p.70

CMS features relevant to design and development:

SEOPage/Screen ManagementSocial Media Integration Analytics ToolsWorkflow ManagementPublishing Controls Security FeaturesVersioning & RollbackContent Repositories Open APIsMultilingual Support

Database Design

  • Data dictionary/library — defines all data fields
  • Entity Relationship Diagram (ERD) — shows relationships between entities
  • Normalisation to 3NF — removes redundancy, ensures data integrity
  • Know when to use databases: user management, e-commerce, diagnostics, performance analysis

Change, Maintain & Support Software

PO3 is worth 18 marks (12.5%). This is assessed through Tasks 3a and 3b — gathering and evaluating feedback, responding to change and demonstrating professional support processes. p.79–80

Drivers of Change p.79

Preventive changes (planned):

  • Changes to regulatory requirements
  • Compatibility with new products or technology
  • Changes in business process
  • Release of a new product/service

Corrective changes (reactive):

  • New zero-day vulnerabilities
  • Targeted attack or data corruption
  • System failures

Iterative development:

  • User/client feedback
  • Developments in technology
  • Competition from other organisations
  • Need to improve efficiency or future-proof products

Software Change Management Process p.79

  1. Identify issues/changes during the feedback/review process
  2. Document developments and changes in the project
  3. Communicate with technical and non-technical audiences appropriately
  4. Plan the changes required
  5. Schedule the changes
  6. Regression testing — ensure changes don't break existing functionality
  7. Release updated products — planned or reactive

Supporting Software Users p.80

Communication channels:

Face-to-faceRemote conferencingBlogs Formal reportsTechnical docsRelease notes User guides / FAQsUser demosScreencast videos API documentation

CONTENT AREA 8.3

Maintaining Code in a Team

Good coding maintenance practices are assessed throughout your portfolio — not just at the change management stage. Every piece of code you write demonstrates your professionalism. p.80

Good Coding Principles for Maintenance

  • Separating code and use of modularity — functions and classes with single responsibility; easier to update individual parts without breaking others
  • Readability — code should be self-documenting with clear names and structure
  • Common/accepted coding conventions — consistent style that other developers can follow
  • Informative commenting/annotation — explain why, not just what; particularly complex logic sections
  • Updating change logs and other documentation — keep a running record of what changed and why

In Practice — What Examiners Expect to See

  • Comments at the top of each file/module explaining its purpose
  • Inline comments explaining non-obvious logic
  • Functions/methods with clear, descriptive names
  • A change log or commit history that shows iterative refinement
  • Evidence of regression testing after every change
  • Version numbers applied to releases
  • Separate modules for UI, business logic and data access (separation of concerns)

Example: Good vs Poor Code Practice

# POOR: unclear name, no comment, magic number
def calc(x):
    return x * 0.2

# GOOD: clear name, docstring, named constant
TAX_RATE = 0.2  # 20% VAT rate per HMRC 2024

def calculate_vat(net_amount: float) -> float:
    """Calculate VAT at the current tax rate."""
    return net_amount * TAX_RATE
TASKS 3a & 3b

Gathering & Evaluating Feedback

Task 3a gathers feedback (internet permitted) · Task 3b evaluates feedback (NO internet). Both contribute to PO3, PO4, PO5 and PO6. Task 3b requires independent, reflective evaluation under exam-like conditions. p.82

Task 3a — Gathering Feedback

  • Design and deliver a survey or questionnaire to target users
  • Carry out user observation and complete observation records
  • Conduct user testing sessions and record outcomes
  • Gather qualitative feedback (verbal, interviews, focus groups)
  • Collect quantitative data (task completion times, error rates, ratings)
  • Document all feedback systematically — attribute each piece of feedback to a specific user and date
  • Ensure your feedback sample represents the target audience appropriately
  • Cross-reference feedback with your original requirements and acceptance criteria
💡 Use a mix of closed questions (ratings 1–5) for quantitative data and open questions for qualitative insight. At least 3 distinct sources of feedback is good practice.

Task 3b — Evaluating Feedback (No Internet)

  • Analyse and categorise the feedback you gathered in 3a
  • Identify patterns — common themes across multiple users
  • Evaluate feedback against your original requirements and acceptance criteria
  • Identify what worked well and what needs improvement
  • Propose and justify specific improvements to the solution
  • Reflect on the software development process itself — what would you do differently?
  • Communicate findings clearly for both technical and non-technical audiences
  • Consider ethical implications of any proposed changes
  • Reference relevant regulations, standards or best practices
No internet in Task 3b. Prepare your analysis templates and evaluation frameworks during earlier tasks so you can work efficiently under restricted conditions.

What Strong Evaluation Looks Like

  • Specific: "3 out of 5 users could not locate the export button" not "some users had issues"
  • Justified: "This suggests the information hierarchy fails the visual hierarchy principle (UX design)"
  • Actionable: "I would redesign the nav bar to move export to the primary action area"
  • Reflective: "If I repeated this project I would conduct usability testing earlier..."
