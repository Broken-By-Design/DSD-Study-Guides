Exam Strategy

STRATEGY
🎯 Walk in prepared. The ESP is about applying what you know under pressure. These strategies help you maximise marks in every task.

How to Break Down Any Question

Step 1 — Read the brief TWICE

First read: get the big picture. Second read: underline every requirement, constraint, and keyword. The brief contains everything you need — the answers are in there.

Step 2 — Spot the command words

These tell you exactly what depth of answer is expected. "Describe" ≠ "Justify". Using the wrong depth wastes time and loses marks.

Step 3 — Plan before you write

Spend 2–3 minutes noting what your answer will cover. This stops you going off-topic and helps you hit every marking point systematically.

Step 4 — Justify everything

Never just state what you did — explain WHY. "I used a dictionary because key-value pairs allow fast lookup by key, which is needed when retrieving user records by ID."

Command Words Decoded

WordWhat it meansExample response style
DescribeSay what something is or does — no opinion needed"A Gantt chart shows tasks, durations, and dependencies across a timeline."
ExplainSay what AND why — add reasoning"A Gantt chart is used because it visually shows which tasks overlap and who is responsible."
JustifyGive reasons AND weigh against alternatives"I chose Agile over Waterfall because requirements may change. Waterfall's rigid phases would not suit this because…"
EvaluateJudge success against criteria, give evidence, suggest improvements"The solution meets criterion X as shown by test Y. However, Z could be improved by…"
IdentifyName it — brief, no explanation needed"The error is a logic error on line 12."
AnalyseBreak it down, examine components, draw conclusions"Breaking down the requirements, the system needs three sub-systems: login, data processing, and reporting."

Functional vs Logical Errors — Know the Difference

🔧 Functional Error

The feature doesn't work at all as specified. The requirement exists, but the code either doesn't implement it or implements something completely wrong.

Example: The brief says "users must be able to log out." The logout button exists but clicking it does nothing — the session is never cleared.

How to find it: Functional testing — run through each requirement and check it works as specified.

🧠 Logical Error

The feature runs without crashing but produces the wrong result. The code is syntactically valid but the logic is flawed.

Example: A discount function that should give 10% off calculates price * 0.9 then subtracts from price again — giving 90% off instead.

How to find it: Use valid and boundary test data — trace the expected output vs actual output step by step.

All Four Error Types at a Glance

Error Type Summary

  • Syntax error — Breaks language rules. Won't even run. (Missing colon, wrong indentation, mismatched bracket)
  • Logic error — Runs but gives wrong output. (> instead of >=, wrong formula, loop off-by-one)
  • Runtime error — Crashes during execution. (Division by zero, index out of range, calling undefined function)
  • Requirement error — Runs, no crash, output seems correct, but doesn't meet the spec. (Wrong output format, missing feature, incorrect validation rule)

How to Spot Them in Task 2

  • Read the requirements FIRST — know what the code SHOULD do before looking at what it does
  • Trace through line by line with sample input — track variable values manually on paper
  • Check boundary conditions — where does the logic change? That's where errors hide
  • Test with all 5 data types — valid, valid extreme, invalid, invalid extreme, erroneous
  • Read error messages carefully — they tell you the line number and type
Exam tip: In Task 2, identify the error type BEFORE fixing. Write in your test table: "This is a logic error because the program runs but produces [wrong value] instead of [expected value] when [input]." Examiners award marks for correct classification.

Fixing the Code

TASK 2 · TESTING
The solution: Correct errors and add/remove code to ensure the given code is fully functional and meets the stated requirements.

Common Defect Types

Syntax Errors

Code that violates the rules of the language (e.g. missing colon, wrong indentation in Python, mismatched brackets). The program won't run at all.

Logic Errors

Code that runs but produces wrong output (e.g. > instead of >=, wrong calculation, loop runs too many/few times).

Runtime Errors

Code that crashes during execution (e.g. dividing by zero, accessing an index that doesn't exist, calling a function that doesn't exist).

Requirement Errors

Code that runs without errors but does not meet the specification (e.g. wrong output format, missing validation, incorrect feature).

Good Fixing Annotations Include

💬
Before: Quote the original buggy line or section. After: Show your corrected version. Why: Explain your reasoning — "I changed < to <= because the boundary value 0 should be accepted as valid."
Defensive programming tip: When fixing, also consider edge cases. Add input validation, exception handling, and meaningful error messages where the original code lacked them.

Task 3: Designing a Solution

TASK 3 · DESIGN
🎨 AO2 + AO3. You break down the problem, design algorithms, follow conventions, and produce documentation detailed enough for a third party to build the solution.

Three Design Requirements

1. Decomposition

Break the problem into smaller computational parts. Use abstraction, show reusable components, identify all sub-systems.

2. Algorithm Design

Describe each part using algorithms. Must include sequence, selection and iteration. Use flowcharts or pseudocode with correct conventions.

3. Design Communication

Produce documents clear enough for the client to make decisions AND for a third party to build the solution from.

Design Tools You Might Use

Flowcharts (BCS symbols) Pseudocode Wireframes ERD / Data Dictionary UML Class Diagrams Data Flow Diagrams
💡 Key: Use a combination of written and diagrammatical presentation. Pure text or pure diagrams alone will limit your marks.

How to Approach Task 3

TASK 3 · DESIGN
🗺️ Task 3 is about structured thinking, not coding. Follow this step-by-step process to produce documentation that earns marks at every level.

Step-by-Step Approach

Step 1 — Extract requirements from the brief

List every "must" and "should" from the brief, numbered. These become your sub-systems. Ask: "What must the system be able to DO?" Each answer = a sub-system to design.

Step 2 — Decompose the system

Draw a hierarchy: System → Sub-systems → Functions.
E.g. Library App → User Login, Book Search, Checkout, Returns → validate_user(), search_by_title(), etc.

Step 3 — Design algorithms for each sub-system

For EACH sub-system: draw a flowchart (BCS symbols) AND write pseudocode. Both must show: sequence, selection (IF/ELSE), and iteration (FOR/WHILE).

Step 4 — Define your data structures

What data does the system store and process? Plan variables, lists, dictionaries, file structures or database tables. Name them clearly — this directly feeds your Task 4a code.

Step 5 — Design the UI / wireframes

Sketch what the user sees at each stage. Label input fields, buttons, output areas. Show navigation between screens. Keep wireframes annotated — explain design choices.

Step 6 — Justify every decision

For each design choice, write WHY. "I used a dictionary rather than a list because lookups by key are O(1) and the system needs to retrieve records by user ID quickly."

What the Examiner Checks For

Mark CriterionWhat you need to showCommon mistake
DecompositionClear hierarchy — system → sub-systems → named functionsOnly listing sub-systems without breaking into functions
Algorithm designBOTH flowchart AND pseudocode for each sub-system; all three constructs visibleOnly doing flowcharts OR only doing pseudocode
Data structuresNamed variables, data types specified, relationships shown (ERD for databases)Just writing "a list" without naming it or specifying what it holds
Third-party buildableCould a stranger build it from your docs alone? If not, add more detailRelying on the examiner to "just know" how something works
Technical vocabularyParameter, return value, iteration, Boolean, integer, key-value pair, etc.Writing in plain English without any technical terms
💡
The golden rule: For every requirement in the brief, your Task 3 docs should show HOW your design meets it. Number your requirements and reference them — "This validate_login() function addresses requirement 3: the system must validate user credentials before granting access."

Flowchart Shapes & Meanings

TASK 3 · DESIGN
⚠️ Use BCS standard symbols only. Using incorrect shapes is a common cause of lost marks in Task 3. Every shape has a specific meaning — learn them.

BCS Standard Flowchart Symbols

Shape NameAppearanceMeaning & When to Use
TerminatorRounded rectangle / Oval (pill shape)START or END of the flowchart. Every flowchart must have exactly one START and at least one END.
ProcessRectangle (plain box)A computation or action — e.g. total ← price × quantity or display welcome message
DecisionDiamond (rhombus)A yes/no question or conditional branch — e.g. Is score >= 50?. Must have exactly two exits: YES and NO.
Input / OutputParallelogram (slanted box)Data entering or leaving the system — e.g. INPUT name or OUTPUT result
Flow ArrowArrow / Line with arrowheadShows the direction of flow between steps. Must always have an arrowhead.
ConnectorSmall circle with a letter/numberA reference point used when the flowchart continues elsewhere on the page (avoids crossing lines).
Sub-processRectangle with double vertical lines on sidesCalls a separate named process / subroutine that is defined elsewhere.
DocumentRectangle with wavy/curved bottomRepresents a document or report produced or consumed by the process.

Key Rules

🔄 Sequence, Selection & Iteration

  • Sequence: boxes flow top-to-bottom in order
  • Selection: diamond with YES/NO branches
  • Iteration: an arrow looping back to an earlier step

⚠️ Common Mistakes

  • Using rectangles for decisions (must be diamond)
  • Forgetting START or END terminators
  • Arrows without arrowheads or flowing backwards
  • Diamond with more than two exits

✅ Mark-earning Habits

  • Label both exits of every diamond (YES / NO)
  • Ensure every path ends at a terminator
  • Keep flow going left-to-right or top-to-bottom
  • Use sub-process boxes for repeated logic
Pseudocode vs Flowcharts: Both must be present in Task 3. Flowcharts are great for showing decision branches and iteration loops visually. Pseudocode is better for showing precise variable names and data types. Use both together for maximum marks.

Pandas — Data Manipulation

What Pandas is for

  • import pandas as pd

How to Break Down a Data Question — Step by Step

1. Identify the data source

Is it a CSV? JSON? Database? Manual input? → Use pd.read_csv() for files, sqlite3 for databases, or build a list/dict from user input first.

2. What processing is required?

Filter? Sort? Aggregate? Count? Group? Use Pandas operations. Break it into steps: load → clean → filter → process → output.

3. What must the output look like?

Printed table? Graph? Saved file? If the brief says "visualise" → use Matplotlib. Always label axes and add a title — unlabelled graphs lose presentation marks.

📝 Worked Mini-Example — Brief: "Load sales data and display a bar chart of total sales by region"

Step 1 — Load: df = pd.read_csv('sales.csv')
Step 2 — Process: by_region = df.groupby('region')['sales'].sum()
Step 3 — Visualise: plt.bar(by_region.index, by_region.values)
Step 4 — Label & display: plt.title('Sales by Region') · plt.xlabel('Region') · plt.ylabel('Total Sales (£)') · plt.show()

Mark tip: Always import both libraries at the top of your file. Add a comment explaining what each Pandas operation does. Examiners reward annotated, readable code — not just code that runs.

System vs User Requirements

TASK 4B · EVALUATION
📋 The ESP brief contains explicit requirements. Task 4b asks you to evaluate against them. You need to know what type each requirement is and how to write evidence for each.

The Two Types of Requirements

👤 User Requirements

What the user needs the system to do, written in plain English from the user's perspective. No technical jargon. Focus on the WHAT, not the HOW.

Characteristics: Written as user stories or plain statements. Usually start with "Users must be able to..." or "The system must allow..."

  • "Users must be able to log in with a username and password"
  • "The system must display results within 3 seconds"
  • "Staff must be able to generate a monthly sales report"
  • "The app must work on both mobile and desktop"

⚙️ System Requirements

The technical specification of HOW the system achieves the user requirements. Written for developers. Must be precise and measurable.

Characteristics: Specify the technical implementation. More precise. Written as technical constraints.

  • "Passwords shall be hashed using bcrypt before storage"
  • "API responses must be returned in under 500ms under normal load"
  • "Reports shall be exportable as CSV and PDF"
  • "The UI shall be built with responsive CSS and tested at 320px, 768px, 1280px"

Functional vs Non-Functional Requirements

✅ Functional Requirements

WHAT the system must do — specific features and behaviours. Testable by checking "does this feature exist and work?"

  • The system must allow users to register an account
  • The system must calculate order totals correctly
  • The system must send a confirmation email on purchase
  • The system must validate that ages are between 0 and 120

Evaluate by: "Does the feature exist and produce the correct output?"

📏 Non-Functional Requirements

HOW the system performs — quality attributes that cut across all features.

  • Performance: Response time, speed under load
  • Security: Input validation, data protection, hashing
  • Usability: Ease of use, accessibility, clear messages
  • Reliability: Uptime, error recovery, exception handling
  • Maintainability: Clear code, comments, modular design

Evaluate by: "How well does it perform against these quality criteria?"

How to Write Your Evaluation Against Requirements

Evaluation Structure

  • 1. List each requirement from the brief (by name or number)
  • 2. State whether your solution meets it — fully, partially, or not
  • 3. Provide specific evidence (test results, screenshots, code reference)
  • 4. If not fully met — acknowledge it honestly and propose the specific fix

Sentence Templates

  • "User requirement [X] stated that [requirement]. My solution meets this because [specific evidence from testing]."
  • "The functional requirement for [X] is fully met — test [Y] confirmed that [result]."
  • "Non-functional requirement [security] is partially met — input is validated but passwords are stored as plaintext. To improve this I would implement bcrypt hashing."
Key exam insight: The ESP brief contains explicit success criteria — these ARE your requirements. Reference them by number or exact wording in your evaluation. Don't write generally about "user needs" — quote the actual criterion from the brief and prove you've met it with evidence.

Task 4b: Reflective Evaluation

TASK 4B · EVALUATION
📝 AO5b — Review (9% of total marks). Apply reflection and evaluation techniques. Show you can honestly assess your own work and identify improvements.

Three Parts of the Evaluation

1. Evidence of Success

Demonstrate that your product meets the brief requirements. Include specific measures against the success criteria listed in the brief.

2. User Needs

Provide evidence from testing that the product meets user needs. Reference actual test results and user feedback where possible.

3. Improvements

Discuss how the solution could be improved if you revisited the problem. Be specific — say what you would do differently and why.

High-Mark Evaluation Language

✅ Do Say

  • "The product meets criterion X because [evidence]…"
  • "Testing showed that when [input], the output was [result], which confirms…"
  • "If I revisited this, I would improve [X] by [specific action] because…"

❌ Avoid

  • Vague statements: "I think it went well"
  • Listing features without linking to the brief
  • "I would add more features" without specifics

Sentence Starters by Section

✅ Evidence of Success

  • "My solution meets criterion [X] because when [action], the system [outcome]…"
  • "I validated the requirement that [X] by running test [Y], which produced [result]…"
  • "The brief stated [requirement]. My solution addresses this by [specific feature]…"

👤 User Needs Evidence

  • "User testing showed that [user type] was able to [task] in [time/steps]…"
  • "Feedback indicated that [feature] was clearly understood because [evidence]…"
  • "The interface met accessibility expectations by [specific design choice]…"

🔧 Improvements

  • "If I revisited this project, I would improve [X] by [specific action] because this would [benefit]…"
  • "A limitation of my current solution is [X]. To resolve this I would [action]…"
  • "With more time I would have implemented [feature] because [justification]…"

📊 Evaluation Checklist

  • ☐ Every success criterion from the brief addressed
  • ☐ Specific test evidence referenced (not vague)
  • ☐ At least 3 distinct, specific improvements named
  • ☐ No phrases like "I think it went well"
  • ☐ Technical vocabulary used throughout
Reflection vs Evaluation: Reflection asks "what happened?" — Evaluation asks "how well did it meet requirements and what could be better?" Aim for evaluation, not just description.
